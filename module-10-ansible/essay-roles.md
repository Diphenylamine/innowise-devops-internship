# Зачем нужны Ansible Roles

Если писать весь playbook в одном файле, проблема вылезает быстро. Допустим, есть инфра с Nginx, PostgreSQL и Redis, и её нужно развернуть на десяти разных проектах. Скопировал playbook десять раз - и при первом же баге исправлять придётся в каждой из десяти копий вручную. Это медленно и неудобо.

Roles решают эту проблему - выносят логику в отдельные переиспользуемые директории. На примере роли nginx структура такая:

- `roles/nginx/tasks/main.yml` - сюда переезжают таски "Update apt cache", "Install Nginx", "Configure Nginx", "Start Nginx"
- `roles/nginx/handlers/main.yml` - handler "Restart Nginx", который срабатывает только при изменении конфига
- `roles/nginx/templates/nginx.conf.j2` - Jinja2-шаблон конфига
- `roles/nginx/vars/main.yml` - переменные по умолчанию, например `nginx_port`

После такого разделения главный playbook становится коротким - это просто список ролей, без детальных тасков:

```yaml
- hosts: webservers
  roles:
    - nginx
    - postgres
```

Главный плюс - переиспользование. Роль `nginx` подключается в любом количестве playbooks и проектов без копирования кода тасков. Нашёл баг или решил поменять конфигурацию веб-сервера - правишь в одном месте, внутри роли, и изменение применяется везде где она используется.

По сути Roles делают для Ansible то же, что функции делают в обычном программировании - убирают дублирование и превращают код в переиспользуемые блоки.
