# Почему Ansible, а не Bash-скрипты?

## Проблема

При масштабировании инфраструктуры Bash-скрипты становятся неуправляемыми по трём причинам:

Не идемпотентны. Повторный запуск `echo "127.0.0.1 db" >> /etc/hosts` добавит строку пятый раз. Чтобы этого избежать — пишешь проверки, код раздувается.

Императивны. Описываешь как сделать, а не что хочешь получить. При ошибке на шаге 3 из 10 скрипт либо упадёт, либо продолжит выполнение в неконсистентном состоянии.

Не масштабируются. Один скрипт — один сервер. На разных дистрибутивах (apt vs yum) нужны разные скрипты.

## Решение — Ansible

Идемпотентность — ключевое свойство. Первый запуск: `changed: true` — пакет установлен. Второй запуск: `changed: false` — Ansible проверил состояние, ничего делать не нужно. Повторный запуск безопасен всегда.

Декларативность — описываешь желаемое состояние: `state=present`, `state=started`. Ansible сам определяет нужные команды под конкретную ОС.

Agentless — работает по SSH. На управляемом сервере нужен только Python.

Inventory — один playbook применяется к сотням серверов через файл инвентаря.

## Вывод

Bash — для простых одноразовых задач. Ansible — для управления конфигурацией в production где важна воспроизводимость и безопасность повторных запусков.
