# Multi-stage builds

## Проблема
В контейнеризации размер образа имеет критическое значение. При создании Docker-образов по принципу «всё в одном», приводит к появлению огромных, неповоротливых контейнеров. Это плохо по трем причинам:
1. **Скорость:** Большие образы долго собираются, медленно загружаются в реестр (Docker Registry) и долго скачиваются на целевые серверы. Это замедляет процесс развертывания (CI/CD) и масштабирования приложений.
2. **Стоимость:** Хранение терабайтов раздутых образов в облаке стоит денег.
3. **Безопасность:** Чем больше программ и библиотек установлено в контейнере, тем уязвимее контейнер. Уязвимость в утилите, которая нужна была только для сборки, может стать лазейкой для взлома работающего приложения.

Один из примеров - разработка на Java с использованием Maven. Для сборки Java-приложения, нужен JDK, Maven и код приложения. При сборке Maven скачивает множество зависимостей в локальный кэш (папка `.m2`). Если это будет обычный Dockerfile, итоговый образ может весить 800мб - 1гб.
Но эти все зависимости не нужны для работы готового приложения, нужен только `.jar` фаил и JRE. Если оставим инструменты сборки, мы получаем лишний объём не нужных файлов

## Решение
Решение этой проблемы - Multi-stage builds. Суть решения заключается в изоляции среды сборки и среды выполнения. Первый этап - сборщик (`FROM image AS builder`). На этом этапе берем тяжелый образ со всеми инструментами, копируем исходный код и запускаем сборку. 
Для решения этой проблемы разработчики Docker внедрили механизм **Multi-stage builds** (многоэтапные сборки). Этот подход позволяет использовать несколько инструкций `FROM` в одном Dockerfile, разделяя процесс на логические этапы (стейджи). Контейнер скачивает зависимости и собирает `.jar` файл.

### Пример Dockerfile
```dockerfile
# Этап 1: Сборка (тяжелый образ)
FROM maven:3.8-openjdk-17 AS builder
WORKDIR /app
COPY . .
RUN mvn clean package

# Этап 2: Выполнение (легкий образ)
FROM openjdk:17-jdk-alpine
WORKDIR /app
# Забираем только готовый результат из первого этапа
COPY --from=builder /app/target/myapp.jar ./myapp.jar
CMD ["java", "-jar", "myapp.jar"]
```

В результате Docker отбрасывает промежуточный образ `builder`. Итоговый контейнер содержит только минимальную среду выполнения и само приложение, а его размер сокращается с 800+ МБ до 100-150 МБ.

## Вывод
Использовать multi-stage builds следует всегда, когда:
- **Работаем с компилируемыми языками программирования:** Java, Go, C++, Rust, C#. Во всех этих случаях инструменты для превращения кода в бинарный файл не нужны для его работы.
- **Собираете frontend-приложения:** на первом этапе используется тяжеловесный Node.js для установки npm-пакетов и сборки статики (React, Vue, Angular), а на втором — легкий веб-сервер (например, Nginx), куда с помощью `COPY --from=builder` переносятся только готовые HTML/CSS/JS файлы.
-  **Требуется повышенная безопасность и минимизация размера:** когда каждый мегабайт имеет значение для скорости CI/CD пайплайнов, а политика безопасности запрещает наличие исходного кода и утилит сборки на production-серверах.

Таким образом, Multi-stage builds позволяет уменьшить итоговый вес файла, ускорить сборку и развёртывание сборки и сохранить безопасность убирая не нужные для работы файлы.
